【測試資料庫決策（使用者決議 2026-07-22）】
07_Testing_and_Acceptance_Plan_v1.0 §5 要求 API 整合測試使用「專用測試資料庫」，
但本機沒有 PostgreSQL／Docker。已詢問使用者，決議：
直接使用現有 Supabase 專案作為測試資料庫，直到專案完成為止。
為避免測試資料寫入正式資料庫，backend/tests/conftest.py 改為：
每個測試在一個 DB Connection 的 Transaction 內執行（SQLAlchemy 2.0
`join_transaction_mode="create_savepoint"`），測試中的所有 commit 只影響 SAVEPOINT，
測試結束後整個 Transaction rollback，實測確認 Supabase 各資料表測試後皆為 0 筆殘留。

【範圍補充說明】
需求追蹤矩陣 §4「開發階段對應」表原本沒有把「收藏」（模組 3.9）與「通知 API」
（模組 3.10 的會員端 GET/mark-read 部分）明確分配到任何一個階段（這是矩陣本身的

     外部，選單和燈窗一起被關閉，「是」的 click 事件因此遺失、永遠無法
     真正登出。修正 handleClickOutside，偵測到點擊目標在 .modal-overlay
     內時直接略過。

【上傳圖片破圖問題 － 全站修復】
使用者回報頭像上傳後顯示破圖，追查後發現是全站性問題：後端上傳 API
回傳的是相對路徑（如 /uploads/avatar/2026/07/xxx.webp，掛載在後端 8000
port 根目錄），但前端所有圖片皆直接 <img src=...> 使用，未轉換為後端完整
網址，導致瀏覽器誤向前端自己的 origin（5173）請求該路徑而 404。
新增 frontend/src/api/client.js 的 resolveMediaUrl() 共用函式（絕對網址
直接放行，相對路徑補上後端 origin），並套用到全站約 19 處圖片顯示位置：
會員/團主頭像（ProfilePage x2、AvatarMenu、GroupLeaderProfilePage、
GroupLeaderListPage）、商品圖（ProductCard、ProductDetailPage、
ProductListPage、ProductFormPage、GroupBuyCreatePage、GroupBuyDetailPage、
FavoritesPage、FollowListPage、OrdersPage、OrderDetailPage）、活動圖
（ActivityCard、ActivityDetailPage、ActivityListPage、ActivityFormPage）。
已用 Playwright 實測頭像上傳全流程（含攔截網路請求確認圖片改用完整網址
且回應 200），確認修復有效。
