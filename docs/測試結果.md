## 1. 已完成的靜態複查

逐檔手動審閱所有 Stage 1 程式碼（無法使用直譯器，人工檢查語法、匯入路徑、SQLAlchemy 用法、外鍵建立順序）。

### 1.1 發現並修正的 Bug

| # | 問題 | 影響檔案 | 修正方式 |
|---|---|---|---|
| 1 | SQLAlchemy Enum 欄位的 `server_default` 直接傳入未加引號的字串（如 `server_default="member"`），會產生無效 SQL `DEFAULT member`（缺少單引號，PostgreSQL 會解析失敗） | `app/models/user.py`、`group_leader.py`、`activity.py`、`group_buy.py`、`order.py`（共 6 處：`user_role`、`group_leader_application_status`、`activity_status`、`group_buy_status`、`order_status`、`cancellation_status`） | 改用 `server_default=text("'member'")` 形式，確保產生的 DDL 為 `DEFAULT 'member'` |

### 1.2 已確認無誤的項目

- **18 張資料表**外鍵建立順序（Alembic migration 內的 `CREATE TABLE` 順序）人工核對過，確認每張表建立時，其所有外鍵參照的目標表皆已存在，不會發生「參照到尚未建立的表」的錯誤。
- **Enum 型別重用**：`payment_method`、`contact_platform` 兩個 Enum 物件在 `group_buy.py` 定義後，於 `order.py` 匯入重用（分別對應 `group_order.payment_method_snapshot`、`leader_contact_platform_snapshot`），確認為同一 Python 物件、同一 DB Enum 型別名稱，避免重複 `CREATE TYPE`。
- **Migration 全部改用原生 SQL（`op.execute`）**，逐字對照 `04_Database_Design_v2.1` 的 CHECK / UNIQUE / Index SQL 語句撰寫，非透過 SQLAlchemy ORM 自動生成，降低 ORM 語法與規格不一致的風險。
- **測試檔案本身不需要真實資料庫連線**：`app/core/database.py` 的 `create_engine()` 為延遲連線（lazy），import 階段不會實際連線 PostgreSQL；`tests/conftest.py` 會在載入 `app` 前注入假的 `DATABASE_URL`／`JWT_SECRET_KEY` 環境變數，因此 `test_models_metadata.py`、`test_health.py` 兩份測試理論上可以在**尚未設定真實資料庫**的情況下執行並通過。

### 已知風險（尚未驗證，需實測確認）

1. **Supabase 權限**：`CREATE EXTENSION IF NOT EXISTS pgcrypto;` 與 `pg_trgm;` 在 Supabase 專案中通常允許（兩者皆為 Supabase 支援的 contrib extension），但仍需實際執行確認帳號權限足夠。
2. **`gen_random_uuid()` 相依 `pgcrypto`**：若 Supabase 專案已內建 `pgcrypto`（多數新專案預設已啟用），此步驟可省略但不影響正確性（`CREATE EXTENSION IF NOT EXISTS` 為冪等操作）。
3. **psycopg 3（`psycopg[binary]`）與 Supabase 連線字串格式**：`.env.example` 已提供將 `postgresql://` 改為 `postgresql+psycopg://` 的說明，但尚未實測連線成功。
4. 尚未驗證 `alembic upgrade head` 執行後，資料庫內的 18 張表、Enum、Constraint、Index 是否與 `04_Database_Design_v2.1` 完全一致（可用 `\d+ <table>` 或 Supabase Table Editor 逐一核對）。
