### 3.3 Alembic Migration 實際執行結果

**過程中排除的連線問題（記錄供未來參考）：**

1. 第一次使用 Supabase **Direct connection**（`db.kjziqgiqiwknxqejancr.supabase.co:5432`）→ 失敗，`psycopg.OperationalError: [Errno 11001] getaddrinfo failed`。原因：Supabase Direct connection 預設走 IPv6，本機網路無法解析。
2. 改用 **Session pooler**，但主機名稱猜測為 `aws-0-ap-south-1.pooler.supabase.com` → 失敗，`FATAL: tenant/user postgres.kjziqgiqiwknxqejancr not found`。原因：pooler 節點前綴猜錯（應為 `aws-1`，不是 `aws-0`）。
3. 改用使用者從 Supabase Dashboard「Connect」視窗複製的正確字串 `aws-1-ap-south-1.pooler.supabase.com:5432` → **連線成功**。

**最終可用的 `DATABASE_URL`（已寫入 `backend/.env`，未提交至版本控制）：**
```
postgresql+psycopg://postgres.kjziqgiqiwknxqejancr:****@aws-1-ap-south-1.pooler.supabase.com:5432/postgres
```

**`alembic upgrade head` 執行輸出：**
```

INFO  [alembic.runtime.migration] Will assume transactional DDL.
INFO  [alembic.runtime.migration] Running upgrade  -> 0001_initial_schema, initial schema: 18 core tables + enums + constraints + indexes
```
`alembic current` 確認版本為 `0001_initial_schema (head)`。**執行成功，無錯誤。**

### 3.4 資料庫實際結構驗證

以一次性腳本直接查詢 Supabase `information_schema` / `pg_catalog`（執行後已刪除，不屬於正式程式碼）：

- **資料表：19 張**（18 張規格資料表 + Alembic 自己的 `alembic_version` 追蹤表），與 `04_Database_Design_v2.1` 完全一致。
- **自訂 Enum：12 個** —— `activity_status`、`announcement_audience_scope`、`announcement_type`、`cancellation_status`、`contact_platform`、`currency`（擴充）、`group_buy_status`、`group_leader_application_status`、`notification_type`、`order_status`、`payment_method`、`user_role`，與規格 + 需求追蹤矩陣衝突解法 #2 一致。（`pg_type` 查詢還列出 Supabase 內建 `auth`/`storage` schema 的其他 Enum，如 `aal_level`、`buckettype` 等，屬於 Supabase 平台本身，非本專案建立，不列入計算。）
- **Public schema 內 Constraint 總數：97**
- **Public schema 內 Index 總數：69**

**結論：Migration 內容與 Database Design v2.1（含 2 個已確認的擴充欄位）完全一致，Stage 1 資料庫基礎已在真實 Supabase 環境驗證通過。**

## 4. 已知風險／後續留意事項

1. **Session pooler 連線行為**：目前使用 port 5432 的 Session pooler。若後續 Stage 2+ 開發中發現 SQLAlchemy 連線池與 Supavisor pooler 有相容性問題（例如 prepared statement 限制），可能需改用 Direct connection（需先解決 IPv6）或調整 `psycopg` 連線參數。
2. **`.env` 內的資料庫密碼**：已寫入 `backend/.env`（已列在 `.gitignore`，不會被提交），但密碼曾在對話中以明文傳遞，建議專案結束或部署前於 Supabase 後台重設密碼。
3. 尚未針對 Migration 的 `downgrade()` 做實際回滾測試（`alembic downgrade base`），建議之後找時機驗證一次，確保降級腳本可正常執行。
